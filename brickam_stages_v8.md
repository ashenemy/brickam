# Brickam — Stages (v8) · план работ для Claude Code

> Используй вместе с `brickam_foundations_v8.md` (архитектура, бизнес-правила, модель данных, настройки). Иди **по одному стейджу за итерацию**. Каждый стейдж: Цель · Задачи · Тесты · Чек-лист готовности. Стейдж считается закрытым только при всех ✓. Тесты обязательны в каждом стейдже (политика — Foundations §18). Коммиты — изолированно по скоупу.

Приоритет потоков: **main-web → сид → vendor-web → admin-web → CI/CD → mobile.**

---

## STAGE 0 — Каркас и DevOps-фундамент
**Цель:** рабочий монорепо с правилами, на котором всё последующее «само» соблюдает стандарты.
**Задачи:** Nx-воркспейс (layout `packages/`); apps `main-web`/`vendor-web`/`admin-web`/`server`; Nx tags+`enforce-module-boundaries`; Biome+ESLint (4 пробела, одинарные кавычки) по матрице; lefthook/husky + commitlint (scope=проект, guard одного проекта); `config-kit` (TOML+env, валидация, `AppConfigService`, рантайм-конфиг фронтов); `.env.example`; `core-kit` (AppException/ErrorCode/Result); `server-kit` (interceptors, AllExceptionsFilter, ValidationPipe, PaginatedDto, composite-декораторы, BaseCrudService/Controller); `db-kit` (Mongo, BaseSchema, BaseRepository<T>); Swagger + генерация `api-kit`; `i18n-kit`; `infra` (Dockerfile.*, docker-compose: Mongo+Redis+server+3 web).
**Тесты:** юнит на config-валидацию, BaseRepository, обёртку ответов, AllExceptionsFilter; смоук-e2e `/api/health`.
**✓ Готовность:** монорепо собирается; `biome check` и `eslint` зелёные, без конфликтов; границы Nx не нарушаются; коммит из 2 проектов отклоняется хуком; Swagger открыт на `/api/docs`; `docker-compose up` поднимает стек; health отвечает; тесты зелёные, покрытие ≥80%.

## STAGE 1 — Дизайн-система, перенос React-UI-кита, каркасы приложений
**Цель:** единые токены, `ui-kit` на Angular+Material+Tailwind (перенесён с React, без багов, адаптивный), оболочки трёх приложений.
**Задачи:** `design-tokens` (tokens.json→tokens.css; Tailwind+Material+mobile из одного источника; токены извлечь в т.ч. из React-кита); **перенос React-UI-кита → Angular** (Foundations §5а): каждый компонент → standalone (signals/`input`/`output`/`ng-content`), Material где есть аналог, Tailwind через токены, **исправить ошибки и респонсивность**, a11y, 3 языка; шеллы навигации `main-web`/`vendor-web`/`admin-web`; переключатель языков hy/ru/en; gate входа по роли. Сверка с Figma (приоритет Figma).
**Тесты:** компонент-тесты каждого перенесённого компонента (рендер, inputs/outputs, состояния); тест, что смена токена меняет и Tailwind-, и Material-значение; тесты адаптивности (брейкпоинты, отсутствие overflow); тест переключателя языка.
**✓ Готовность:** React-кит полностью перенесён на Angular+Material+Tailwind, баги исходника устранены, компоненты адаптивны и доступны, без хардкод-цветов (всё через токены); 3 приложения запускаются с общими токенами; правка токена отражается везде; язык переключается; расхождения с Figma зафиксированы; тесты/покрытие ок.

## STAGE 2 — Auth и пользователи
**Цель:** регистрация/вход по телефону+паролю+OTP, роли.
**Задачи:** `feature/auth`+`feature/users`; регистрация `+374…`+пароль+роль→OTP(SMS, мок)→активация (вендор `pending`); вход (+OTP для нового устройства); сброс через OTP; JWT access+refresh; guards/`PermissionsGuard`; параметры OTP из конфига, хеш кода, rate-limit; secure storage/httpOnly.
**Тесты:** юнит OTP (генерация/TTL/попытки/кулдаун/хеш); e2e регистрация→OTP→вход→refresh; негативы (неверный OTP, просрочка, rate-limit); проверка ролевых guard'ов.
**✓ Готовность:** полный цикл auth работает на моке SMS; токены/refresh корректны; OTP-лимиты соблюдаются; роли гейтят доступ; тесты/покрытие ок.

## STAGE 3 — Шаблоны и уведомления
**Цель:** тексты писем/SMS не в коде с самого начала.
**Задачи:** `feature/templates` (email/sms, мультиязычные, плейсхолдеры Handlebars/Liquid, белый список переменных, версия); `feature/notifications` тянет тело из templates; OTP-SMS — через шаблон; дефолтные шаблоны.
**Тесты:** юнит рендера шаблона с подстановкой и валидацией переменных; тест, что OTP/уведомления берут текст из templates, а не из строк.
**✓ Готовность:** ни одного хардкод-текста уведомлений; шаблоны рендерятся на 3 языках; неизвестная переменная отлавливается; тесты/покрытие ок.

## STAGE 4 — Каталог + SEO
**Цель:** товары (продажа), обложка фото/видео, поиск/фильтры, SEO.
**Задачи:** `feature/catalog` (категории, товары CRUD-бэкенд, `cover.mediaType` фото/видео, gallery, скидка товара, attributes); поиск/фильтры (категория, цена, поставщик, рейтинг, наличие, регион), сортировка, серверная пагинация; карточка и листинг на `main-web`; видео-обложка (автоплей muted loop/нативный плеер, thumbnail-фолбэк); валидация медиа по `platform_settings.media`; **SEO:** Angular SSR-сервис, edge-детект бот-UA (список из настроек)→SSR, люди→CSR; пререндер каталожных/товарных маршрутов; meta/OG, `sitemap.xml`/`robots.txt`.
**Тесты:** юнит фильтров/пагинации/расчёта цены со скидкой; e2e поиск/фильтр/карточка; тест видео-vs-фото обложки; тест маршрутизации бот→SSR, человек→CSR; тест валидации媒 по лимитам.
**✓ Готовность:** каталог ищется/фильтруется/пагинируется; обложка-видео работает с фолбэком; бот получает SSR-HTML, человек — CSR; sitemap/robots отдаются; лимиты медиа применяются; тесты/покрытие ок.

## STAGE 5 — Корзина, заказ, платёж с разбивкой, комиссия
**Цель:** оформление и один платёж, разбитый по вендорам, комиссия с продавца.
**Задачи:** `feature/orders`+`feature/payments`; мультивендорная корзина (группировка по вендору); оформление (адрес, применение скидки лояльности позже в Stage 12 — пока хук); один платёж (заглушка провайдера)→`payments.splits`; `vendor_orders` с расчётом **комиссии на сумму после скидки в AMD**, `payout`; статусы заказа и доставки (`deliveryEvents`); номера заказов.
**Тесты:** юнит расчёта: subtotal→комиссия (после скидки, округление AMD)→payout→splits; e2e оформление→оплата(заглушка)→создание vendor_orders; мультивендорный кейс; негативы (пустая корзина, нет товара/остатка).
**✓ Готовность:** один платёж корректно разбивается по вендорам; комиссия = после скидки, целые AMD; payout верный; статусы меняются; тесты/покрытие ок.

## STAGE 6 — Отложенные (вишлист)
**Задачи:** `feature/wishlist`; сердечко на карточках/в карточке; страница «Отложенные»; «в корзину»; бейдж-счётчик.
**Тесты:** юнит add/remove/idempotency; e2e добавление→страница→в корзину.
**✓ Готовность:** товар откладывается/убирается, переносится в корзину, счётчик верен; тесты/покрытие ок.

## STAGE 7 — Отзывы и рейтинги
**Задачи:** `feature/reviews`; отзыв после завершённого заказа (1–5+текст); пересчёт `ratingAvg/ratingCount` вендора/товара; отображение в карточке/профиле; модерация-флаг.
**Тесты:** юнит пересчёта рейтинга; e2e «только после завершённого заказа»; защита от повторного отзыва.
**✓ Готовность:** отзыв оставляется лишь по завершённому заказу; рейтинги пересчитываются; тесты/покрытие ок.

## STAGE 8 — Чат
**Задачи:** `feature/chat`; realtime Socket.IO (Gateway), список диалогов, непрочитанные, статусы прочтения, текст+файлы; покупательский UI в `main-web`.
**Тесты:** юнит unread/доставки; e2e обмен сообщениями двух ролей; авторизация участников канала.
**✓ Готовность:** реалтайм-чат работает, непрочитанные/прочтения корректны, доступ только участникам; тесты/покрытие ок.

## STAGE 9 — Кастомные инвойсы
**Задачи:** `feature/invoices`; продавец (право `invoices.create`) формирует инвойс покупателю в диалоге (позиции, скидка, срок); сообщение-инвойс в чат; оплата→заказ (через Stage 5); генерация PDF; статусы draft|sent|paid|expired|cancelled. Вендор-UI создания — в `vendor-web`.
**Тесты:** юнит итогов/просрочки инвойса; e2e создание→оплата→заказ; проверка права `invoices.create`.
**✓ Готовность:** инвойс создаётся продавцом, приходит в чат, оплачивается→заказ; PDF генерируется; права соблюдены; тесты/покрытие ок.

## STAGE 10 — Калькуляторы
**Задачи:** `calc-kit` (`BaseCalculator`+конкретные: краска, плитка, напольные, обои, бетон/стяжка, гипсокартон, штукатурка/шпаклёвка); результат→подбор товаров категории→в корзину/отложенные; мультиязычные подписи; UI в `main-web`.
**Тесты:** юнит формул каждого калькулятора (граничные значения, запас%); e2e «посчитал→подобрал товары→в корзину».
**✓ Готовность:** все калькуляторы считают верно и ведут к товарам; тесты/покрытие ок.

## STAGE 11 — Валюта и конвертация
**Задачи:** `feature/currency`; база AMD; переключатель отображения; курсы из провайдера (ЦБ Армении+фолбэк), кеш `exchange_rates`, дневное обновление; расчёты в AMD, отображение — конвертация.
**Тесты:** юнит конвертации/фолбэка/кеша; тест, что расчёты заказа остаются в AMD.
**✓ Готовность:** цены отображаются в выбранной валюте, расчёты в AMD, фолбэк работает; тесты/покрытие ок.

## STAGE 12 — Лояльность (конструктор + применение)
**Задачи:** `feature/loyalty`; модель `loyalty_programs` (basis total_spend|order_count, tiers[порог,скидка]); вычисление текущего уровня покупателя; авто-применение скидки на оформлении (★ несёт платформа — Foundations §11); `loyalty_ledger` и обновление метрики при завершении заказа; UI баланса/уровня в `main-web`. (Конструктор-редактор — в админке Stage 17.)
**Тесты:** юнит выбора уровня по метрике; юнит расчёта скидки (percent/amount, неотрицательность, взаимодействие со скидкой товара и комиссией); e2e «заказ→рост метрики→новый уровень→скидка на след. заказе».
**✓ Готовность:** уровень считается по сумме или числу заказов; скидка применяется; метрика растёт; комиссия вендора не затронута скидкой лояльности; тесты/покрытие ок.

## STAGE 13 — AI-поиск
**Задачи:** `feature/ai-search`+`ai-kit`; эндпоинт: текст задачи→LLM (строгий JSON `{projectType,themes[{name,materialCategories,keywords}]}` со схемой+ретраями)→гибридный подбор (текст+vector по `embedding`)→результат по AI-темам; генерация эмбеддингов при создании/правке товара; кеш частых запросов; UI «Опишите проект» в `main-web`.
**Тесты:** юнит парсинга/валидации JSON-ответа (включая мок-ретраи); юнит сборки гибридного запроса; e2e «описание→темы→товары» на мок-провайдере.
**✓ Готовность:** запрос даёт сгруппированный по темам результат; невалидный ответ LLM обрабатывается ретраем/фолбэком; эмбеддинги генерятся; тесты/покрытие ок (внешний AI — за моком в тестах).

## STAGE 14 — Реалистичный сид
**Задачи:** `tools/seed`; вендоры с реалистичными арм. названиями; дерево категорий; **80–150 товаров** (часть с видео-обложкой), цены AMD, скидки; отзовы/рейтинги; демо-покупатели; примеры чатов с инвойсами; курсы валют; **дефолтные шаблоны писем/SMS/страниц**; **стартовая программа лояльности**; `platform_settings` (media/aiPrompts/seo).
**Тесты:** юнит идемпотентности сида (повторный запуск без дублей); проверка целостности связей.
**✓ Готовность:** одна команда наполняет БД убедительными данными; приложение выглядит готовым к демо; повторный сид безопасен; тесты ок.

## STAGE 15 — Кабинет продавца (vendor-web)
**Задачи:** `feature/vendors`/`vendor-members`/`vendor-bulk`/`analytics`/`subscriptions`; товары (CRUD, обложки фото/видео); **массовые действия** (цена на %/сумму/значение, скидки, остатки, активация/скрытие, смена категории; превью+подтверждение; большие наборы — BullMQ); заказы и статусы доставки; **суб-аккаунты и права** (`PermissionsGuard`); подписка (план без ограничений пока); **объёмная аналитика** (выручка/GMV по периодам, средний чек, воронка статусов, топ-товары по продажам/просмотрам, конверсия, рейтинг во времени, остатки/алерты, экспорт CSV/Excel; aggregation+кеш; scoped).
**Тесты:** юнит bulk-операций (корректность массовой правки, превью); юнит агрегатов аналитики; e2e ролей суб-аккаунтов (member без права не видит/не может); e2e смены статусов заказа.
**✓ Готовность:** продавец ведёт товары и заказы; массовые действия работают (вкл. фоновые); права суб-аккаунтов соблюдаются; аналитика корректна и scoped; экспорт работает; тесты/покрытие ок.

## STAGE 16 — AI-ассистент продавца
**Задачи:** `feature/ai-assistant`; промпт-управляемая генерация (база-промпт из `platform_settings.aiPrompts`+промпт продавца+контекст): описание (LLM, hy/ru/en), картинка (image-провайдер), видео-превью (ffmpeg-слайдшоу/опц. video-API); `ai_jobs` асинхронно с прогрессом; результат→возможная видео-обложка; учёт лимитов медиа.
**Тесты:** юнит сборки промпта из шаблона+ввода+контекста; юнит жизненного цикла `ai_jobs` (queued→done/failed); e2e на мок-провайдерах.
**✓ Готовность:** генерация запускается промптом, идёт асинхронно, результат прикрепляется к товару/обложке; провайдеры за моками в тестах; тесты/покрытие ок.

## STAGE 17 — Админка (admin-web)
**Задачи:** `feature/disputes`/админ-разделы; модерация вендоров/товаров; управление комиссией (через конфиг/настройку) и планами подписок; споры; общая аналитика (GMV, выручка платформы=комиссии, заказы); **редактор шаблонов** писем/SMS/страниц; **конструктор лояльности** (basis+уровни+скидки); **platform_settings**: ограничения видео/изображений, промпт-шаблоны AI, список бот-UA для SEO; аудит-лог.
**Тесты:** юнит сохранения/валидации настроек (media/aiPrompts/seo); e2e редактирования шаблона→рендер с новыми текстами; e2e конструктора лояльности→применение на покупательской стороне; гейт admin-роли.
**✓ Готовность:** админ правит шаблоны, лояльность, медиа-лимиты, промпты, бот-UA — и это влияет на систему без деплоя; модерация/споры работают; аудит пишется; тесты/покрытие ок.

## STAGE 18 — CI/CD (полный) и стейджинг
**Задачи:** GitHub Actions: `biome check→eslint→typecheck→test(порог)→nx affected build→docker build/push (GHCR)→deploy staging`; секреты через CI/секрет-менеджер; health-checks; Sentry; раздельные пайплайны/образы для server и 3 web.
**Тесты:** прогон пайплайна на тестовой ветке; проверка, что падение тестов/покрытия блокирует мерж.
**✓ Готовность:** пуш→авто-проверки+сборка+деплой стейджинга; красные тесты блокируют; образы публикуются; стейджинг отвечает.

## STAGE 19 — Мобильное приложение (NativeScript)
**Задачи:** `apps/mobile`; переиспользование пакетов (`api-kit`/`state-kit`/`auth-kit`/`i18n-kit`/`design-tokens`); нативный UI (`mobile-ui-kit`); auth+OTP, каталог (видео-обложки нативно), корзина/заказ, чат, вишлист, калькуляторы, лояльность; push (FCM/APNs); secure storage.
**Тесты:** юнит переиспользуемой логики на mobile-цели; smoke-навигация ключевых экранов.
**✓ Готовность:** приложение собирается под iOS/Android, проходит ключевые сценарии, использует общие пакеты; push/secure storage работают; тесты ок. *(Стор-публикация/подпись — ручные шаги вне автономии агента.)*

### Демо-MVP срез (если нужна первая презентация быстро)
Минимум для убедительного демо: **STAGE 0–5 + 14 (сид) + базовая часть 15**. Остальное (лояльность, AI, чат/инвойсы, mobile, полная админка) — после. Это даёт работающий каталог с обложками, заказом и платёжной разбивкой на наполненных данных.
